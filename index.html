Ovaflow

Ovaflow

Ovaflow is a light-weight rhythm game fueled by the user community. (similar to step-mania) Currently developed for the Android platform only. The objective of the game is to press the correct sequence of keys or buttons as notes cascade down the screen towards a special buttons on the screen. The game's score will be calculated based on the accuracy of the button presses atop some precision tolerance, and might end prematurely if the user makes too many errors. The differentiating factor is that all the "beatmaps" are created by users which may be uploaded onto the central repository and downloaded along with avatars.

What started as a simple prototype for a class ended as a fully functional game. I was primarily responsible for the Android platform but also built up the initial framework for the server

Technologies

J2EE, Android, OpenGL ES 2.0, REST, Sqlite, MySQL

Design

The project is divided into two parts, the Android platform (Client) and the server repository.

Client

The Client only supports Android devices. The rendering is done by using OpenGL ES 2.0. The rendering is displayed via a SurfaceView. The downloading of songs is done by sending REST-ful requests to the server and using AsyncTasks to fetch the songs over a stream.

Server

The server provides the user high scores, avatars, and a song repository accessed via JAX-RS

Chitchat

Chitchat

Chit-chat is an interactive chat-room where participants can talk to each other. Aside from being able to talk to each other, they can also walk and jump around in their avatars which can be seen by other participants.

This started as a one day/night experiment where I wanted to learn about Java Websockets as well as WebGL.

The sprites are not drawn by me and will be replaced upon request from the original author. This is a simple exerpiment and I do not intend to infringe on any copyright.

Technologies

J2EE, Websocket, WebGL, ThreeJS, HTML5, CSS3

Design

The web application is developed using websockets using javascript to connect to a Java Websocket service.

Client

The client consists of HTML5 components. Javascripts using WebGL as well as AJAX calls to the websocket. JSON is (temporarily) used to convey all messages. (In the future, I would like to send encoded byte streams)

Server

The server uses Java Websockets to relay messages to all participating parties by encapsulating them through JSON request/responses

Pictoview

Picture Viewer Repository

Pictoview is a central photos (and later songs/videos) repository. It allows viewers to access their photos in an organized and easy to use manner. It includes the ability to add customized tags to albums and later search upon those tags. The app allows users to store images in a centralized database without having to store the data on their devices.

It started as a means for me to store my large repository of pictures and organize them in a manageable fashion without having to store all the pictures on my mobile devices.

Technologies

J2EE, SpringMVC, REST, EJB, Sqlite, HTML5, CSS3, Javascript, jQuery, Android

Design

The project is divided into three Components, the client, the request server, and the Enterprise Java Bean.

Client

Android

The Android client uses volley to send HTTP requests to retrieve informations & photos from the server using REST-ful API. (Currently in development)

Browser

The browser is partially generated HTML page via JSP, with dynamically generated content via Ajax calls. Uses jQuery to process client response data and page dynamics.

Server

Spring

Uses Spring MVC to generate client JSP pages as well as Ajax JSON responses. Images are send via output streams.

REST

Uses JAX-RS to supply API calls to the client. The information is taken from the enterprise bean.

EJB

The EJB provides access to the database which stores all the images and their information
